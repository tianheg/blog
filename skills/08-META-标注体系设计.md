# 元技能08: 标注体系设计 — 语义标记符号的工程化设计原则

## Agent 执行摘要

**一句话**：设计一套一致、可验证、可扩展的标记符号体系，让机器和人类对同一文本有相同的理解。
**何时使用**：需要为内容添加结构化标记、设计分类标签、建立消歧规则时。
**核心步骤**：
1. 明确标注目的（提取什么信息？用于什么下游任务？）
2. 选择范式：边界标记、类型标记、属性标记、关系标记
3. 设计标记规则：语法、值域、嵌套约束
4. 提供验证模式（正则/JSON Schema）
5. 编写示例和反例

## 检查清单

- [ ] 标注目的明确（不为标注而标注）
- [ ] 标记语法有明确的开始/结束边界
- [ ] 值域约束已定义（允许哪些值？）
- [ ] 有正则或 Schema 可自动验证
- [ ] 提供了正确示例和错误示例
- [ ] 向后兼容（新增标记不破坏旧标记）

## 详细参考

### 四种标记范式

| 范式 | 形式 | 用途 | 示例 |
|------|------|------|------|
| **边界标记** | `【内容】` | 强调特定片段 | `【重点】`、`【警告】` |
| **类型标记** | `#+BEGIN_TYPE` | 代码块、引用 | `#+BEGIN_SRC bash` |
| **属性标记** | `key: value` | 元数据 | `来源: https://...` |
| **关系标记** | `[[target]]` | 链接关联 | `[[git-commit]]` |

### 设计原则

1. **边界明确** — 开始和结束标记必须对称、不可嵌套冲突
2. **值域约束** — 标记的值必须是枚举值或符合正则
3. **向后兼容** — 新增标记类型不破坏已有解析逻辑
4. **可验证** — 能用正则或脚本自动检查合法性

### 本项目的标记实践

**TIL 内联标记**：
```
【重点】 → 需要特别注意的内容
【代码】 → 一行代码或命令
【步骤】 → 操作步骤编号
【警告】 → 可能导致错误的操作
【提示】 → 有用的补充信息
```

**Org Mode 标记**：
```org
#+BEGIN_SRC bash
#+END_SRC

#+BEGIN_QUOTE
#+END_QUOTE
```

### 验证正则示例

```python
# 验证 【标记】格式
import re
pattern = r'【(重点|代码|步骤|警告|提示)】'
```

**核心洞察**：标注体系不是一次性设计，而是随认知深化而迭代演化。先设计最小可用集，再按需扩展。
