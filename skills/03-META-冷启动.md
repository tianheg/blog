# 元技能03: 冷启动 — 从零开始构建AI驱动工序的方法论

**定位**: 指导如何在没有先验知识、没有训练数据、没有成熟SKILL文档的情况下，快速启动一个新的AI驱动任务。

## 一、什么是冷启动

**冷启动**是指在缺乏以下条件时启动新任务的过程：
- 无先验知识（不知道哪些方法有效）
- 无训练数据（没有标注好的样本）
- 无成熟规则（不知道错误模式）
- 无质量基线（不知道"好"的标准）

## 二、冷启动的五个阶段

```
┌─────────────────────────────────────────────────┐
│  Phase 0: 任务定义（Task Definition）            │
│  ├─ 明确要做什么、为什么做、成功标准是什么        │
│  ├─ 定义输入输出格式                             │
│  └─ 耗时: 1-2天                                  │
├─────────────────────────────────────────────────┤
│  Phase 1: 手动处理（Manual Work）                │
│  ├─ 人工处理3-10个样本                           │
│  ├─ 建立直觉、发现边界情况、提炼初步规则          │
│  └─ 耗时: 2-5天                                  │
├─────────────────────────────────────────────────┤
│  Phase 2: SKILL文档编写（Documentation）         │
│  ├─ 将Phase 1的直觉转化为可执行规则              │
│  ├─ 5-15条基础规则+示例                          │
│  └─ 耗时: 1-2天                                  │
├─────────────────────────────────────────────────┤
│  Phase 3: 小规模试点（Pilot）                    │
│  ├─ Agent处理10-20个样本                         │
│  ├─ 对比人工处理，发现规则漏洞                    │
│  └─ 耗时: 1-2天                                  │
├─────────────────────────────────────────────────┤
│  Phase 4: 全量执行（Full Execution）             │
│  ├─ Agent批量处理全部数据                        │
│  ├─ 抽样检查、记录错误模式                        │
│  └─ 耗时: 2-5天                                  │
└─────────────────────────────────────────────────┘
```

**总耗时**: 7-16天完成冷启动

## 三、Phase 0: 任务定义

### 三个灵魂拷问

| 问题 | 内容 | 示例（博客文章生成） |
|------|------|---------------------|
| **要做什么?** | 输入/输出/粒度 | 输入：关键词/大纲；输出：完整文章；粒度：1500-3000字 |
| **为什么做?** | 解决什么问题 | 提高内容产出效率，保持风格一致性 |
| **怎样算成功?** | 定量/定性指标 | 准确率>80%，人工审核通过，读者反馈正面 |

### 输出格式Schema

```json
{
  "title": "string, 文章标题",
  "summary": "string, 100字摘要",
  "content": "string, Markdown正文",
  "tags": ["array, 标签"],
  "category": "enum, 分类",
  "estimated_read_time": "number, 预估阅读时间(分钟)"
}
```

## 四、Phase 1: 手动处理

### 样本选择策略

| 维度 | 策略 | 示例 |
|------|------|------|
| 体裁多样性 | 每种类型至少1个 | 技术/生活/读书笔记 |
| 复杂度梯度 | 简单→中等→复杂 | 短文/标准/深度长文 |
| 边界情况 | 已知的特殊case | 含代码/图片/表格的文章 |

**样本数量**: 3-10个（太少无法发现模式，太多耗时过长）

### 手动处理的四个步骤

1. **盲标（Blind Work）** — 不参考任何规则，凭直觉处理
2. **自我审查（Self-Review）** — 检查粒度一致、类型清晰、无遗漏
3. **提炼规则（Rule Extraction）** — 从处理过程中提炼决策规则
4. **交叉验证（Cross-Validation）** — 用第2-3个样本验证规则

## 五、Phase 2: SKILL文档编写

### SKILL文档的最小可行版本（MVP）

**必备内容（MVP）**:
- ✅ 任务目标（1段）
- ✅ 输出格式（JSON schema）
- ✅ 核心规则（5-10条）
- ✅ 示例（2-3个完整示例）
- ✅ 边界情况（5-10个困难case）

**规则表达的三个原则**:
1. **具体胜于抽象** — 每条规则可直接应用
2. **示例驱动** — 每条规则至少附1个示例
3. **允许模糊** — 冷启动阶段标注"待定"，迭代时决定

## 六、Phase 3: 小规模试点

### 试点规模

- 10-20个样本
- 覆盖Phase 1手动处理的样本（用于对比）
- 新增未见过的样本（用于泛化测试）

### 试点分析的三个维度

1. **与人工处理的对比** — 重叠度、遗漏、多余
2. **输出格式检查** — 是否符合Schema
3. **粒度一致性检查** — 不同样本处理质量是否一致

### 试点后的决策点

- **SKILL文档是否需要修订?** — 若>30%误解规则 → 重写
- **是否进入Phase 4全量执行?** — 与人工处理重叠度>70%才进入

## 七、Phase 4: 全量执行

### 批量执行策略

```
BATCH_SIZE = 5
TOTAL_BATCHES = 26  # 130篇文章 / 5

for batch in 1..26:
    start = (batch-1)*5+1
    end = batch*5
    execute_batch(start, end)
    validate_batch(batch)
    save_results(batch)
    report_progress()
```

### 过程监控

- 实时质量监控（统计错误、检查异常）
- 抽样检查（10-20篇人工复查）
- 错误模式提取（冷启动的核心产出）

## 八、冷启动的成功标准

### 完成的标志

- [ ] 全部数据已处理
- [ ] 抽样准确率 > 60%（允许较低，迭代会提升）
- [ ] 错误模式表已建立（15-30条）
- [ ] SKILL文档已完善（从5条→30条）
- [ ] 质量基线已确定（知道"好"的标准）

**不追求**:
- ❌ 准确率>90%（这是迭代的目标）
- ❌ 完美的SKILL文档（迭代中会持续完善）
- ❌ 零错误（允许系统性错误，后续批量修正）

## 九、常见陷阱

| 陷阱 | 症状 | 解决 |
|------|------|------|
| **过早优化** | Phase 1追求完美，每个样本4-8小时 | 快速完成（2小时/样本），迭代时细化 |
| **样本偏差** | 只选简单样本，复杂场景准确率极低 | 刻意包含简单/中等/复杂样本 |
| **跳过试点** | SKILL后直接全量，发现20%格式错误 | 严格执行试点（10-20个），不通过不进入全量 |

## 十、总结

**冷启动的核心是快速建立认知→形成方法论→批量执行→积累模式**。

- 不追求高准确率（60-80%即可），追求广度覆盖+模式积累
- 核心产出是错误模式表，而非完美数据
- 完成的标志是进入迭代阶段的准备就绪

**时间分配**: Phase 0（1-2天）+ Phase 1（2-5天）+ Phase 2（1-2天）+ Phase 3（1-2天）+ Phase 4（2-5天）= **7-16天完成冷启动**
