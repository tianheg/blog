# 元技能07: 可读性 — 人类优先的数据格式设计

## Agent 执行摘要

**一句话**：数据的第一受众是人类，第二受众才是机器。设计任何输出格式时，先保证人能在 30 秒内理解结构，再考虑机器解析。
**何时使用**：设计 SKILL 输出格式、JSON Schema、数据交换格式时。
**核心步骤**：
1. 选择格式：Markdown 表格 > YAML > JSON > 纯文本
2. 确保自解释：字段名即注释，无需额外说明
3. 层次清晰：用空行和标题分隔逻辑块
4. 提供示例：每个格式配 1 个完整示例

## 检查清单

- [ ] 人类可在 30 秒内理解结构，无需阅读说明文档
- [ ] 字段名自解释（`correction_rate` 而非 `cr`）
- [ ] 有完整的示例（非片段）
- [ ] 层次清晰，逻辑块之间有视觉分隔
- [ ] 机器可解析（有明确的 Schema 或约束）

## 详细参考

### 核心原则

1. **自解释** — 字段名即注释，看到就能猜出含义
2. **结构化** — 用表格、列表、代码块，避免大段自然语言
3. **示例驱动** — 每种格式配至少 1 个完整示例
4. **认知可读** — 层次清晰、视觉分隔、关键信息前置

### 格式选择优先级

| 场景 | 推荐格式 | 原因 |
|------|----------|------|
| 规则/对照表 | Markdown 表格 | 人易扫描，机器可解析 |
| 配置数据 | YAML | 层次清晰，注释友好 |
| 结构化输出 | JSON | 机器标准，人可读性一般 |
| 长文本内容 | Org Mode / Markdown | 人优先 |

### Bad vs Good 示例

**❌ Bad（纯文本，难扫描）**：
```
文件名应该用小写字母，不能有空格，software目录下要用前缀命名。
TIL文件需要包含重述理解和内容主体两部分。
代码块要指定语言。
```

**✅ Good（结构化，易扫描）**：
```markdown
| 检查项 | 要求 | 示例 |
|--------|------|------|
| 文件名 | 小写、无空格 | `git-rebase.org` |
| 目录 | software/ 用前缀 | `software/git-xxx.org` |
| 结构 | 含 💡 重述理解 + 📋 内容主体 | 见模板 |
| 代码块 | 指定语言 | `#+BEGIN_SRC bash` |
```

### 层次分隔规范

```markdown
## 一级标题（主要模块）
内容...

### 二级标题（子模块）
内容...

---

## 下一个一级标题（视觉分隔）
```
