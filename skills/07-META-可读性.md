# 元技能07: 可读性 — 人类优先的数据格式设计

**核心思想**: **"数据的第一受众是人类，第二受众才是机器"**

## 一、核心原则

| 原则 | 说明 |
|------|------|
| **人类优先** | 格式设计首先考虑人类阅读，其次考虑机器解析 |
| **Git友好** | 所有数据文件应易于版本控制（文本格式、行式结构） |
| **调试透明** | 中间结果可视化，黑箱流程文档化 |
| **渐进降级** | 复杂数据提供多层次视图（摘要→详情→原始） |

## 二、双格式输出（JSON + Markdown）

```
kg/events/data/
├── 001_文章_事件索引.md      # 人读: 表格+Markdown
├── event_relations.json      # 机读: 结构化JSON
└── event_relations_summary.md # 人读: 统计摘要
```

### Markdown格式示例

```markdown
# 001 文章标题 - 事件索引

## 概览表格
| 事件ID | 事件名 | 类型 | 主要人物 | 时间 |
|--------|--------|------|---------|------|

## 详细记录
### 001_001 事件名称
- **类型**: 类型A
- **人物**: 人物A, 人物B
- **描述**: 事件描述...
```

### JSON美化格式

```json
{
  "_metadata": {
    "generated_at": "2026-03-17T14:32:10Z",
    "total_items": 100
  },
  "items": [
    {
      "id": "001",
      "name": "名称",
      "_note": "注释说明"
    }
  ]
}
```

## 三、中间输出设计

**原则**: 每个处理阶段保存输出

```
Stage 1: tmp/stage1_output.json + stage1_preview.md
Stage 2: tmp/stage2_output.json + stage2_preview.md
Stage 3: data/final.json + final.md
```

**收益**:
- 出问题可定位具体阶段
- 支持断点续跑
- 方便A/B测试

## 四、代码可读性

### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 函数 | 动词开头，描述性强 | `extract_events`, `validate_data` |
| 变量 | 避免单字母 | `events`, `correction_count` |
| 常量 | 全大写 | `MAX_RETRY_COUNT` |

### 注释原则

```python
def extract_events(text):
    """
    从文本中提取事件
    
    Args:
        text: 输入文本
        
    Returns:
        事件列表
        
    设计决策: 使用正则而非LLM，因为：
    - 速度提升100倍
    - 精度达到95%
    - 详见 doc/design/event_extraction.md
    """
```

## 五、错误消息设计

**❌ 差的错误消息**:
```
Error: check failed
```

**✅ 好的错误消息**:
```
文件: chapter_md/001.md
❌ [L2-格式错误] 第42行
   未闭合的标签: <div class="content"
   建议: 补全闭合符号 → <div class="content">
```

## 六、可视化辅助

- **表格**: 统计数据、对比分析
- **树形图**: 分类体系、层次结构
- **时间轴**: 事件分布、演进过程
- **流程图**: 工作流、决策路径

## 七、总结

**可读性的三个层次**:
1. **语法可读** — 格式美化、符号轻量
2. **语义可读** — 注释详尽、文档完善
3. **认知可读** — 层次清晰、可视化辅助

**核心洞察**: 可读性是所有元技能的基础设施 — 没有可读性，其他方法论都会打折扣。
