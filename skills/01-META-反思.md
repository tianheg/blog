# 元技能01: 反思 — Agent驱动的质量审查通用方法论

**定位**: 适用于内容生成、代码审查、数据标注等所有需要AI生成内容质量审查的工序。本文档提炼项目中所有反思实践的共性方法，指导新工序设计。

## 零、什么是反思（Reflection）

**反思**是对AI生成内容的系统性质量审查与迭代修正，通过Agent工具自动化执行。

**区别于人工审核，反思具备以下特征**:
- **Agent驱动**: 由Claude Agent自主执行审查-修正-验证循环
- **方法论嵌入**: 将领域知识、错误模式、审查标准编码为SKILL文档，注入Agent提示词
- **迭代收敛**: 多轮反思逐步提升质量，每轮修正量递减直至收敛
- **模式积累**: 每轮提取新发现的错误模式，沉淀为可复用知识

### 为什么需要反思?

AI生成内容存在三类典型问题:
1. **系统性偏差** — 如博客文章中将相似主题的内容重复处理（错误模式在全书重复出现）
2. **确定性不足** — 如日期标注本可精确到具体年月，却标注为"约2020年"（保守但不准确）
3. **局部语境误判** — 如同一术语在不同文章中需要不同分类，单次生成缺乏全局视野

## 一、反思的三个维度

本项目的反思实践可归纳为三个正交维度的组合:

```
┌─────────────────────────────────────────────────┐
│  审查粒度（Granularity）                         │
│  ├─ 按篇反思（Chapter-wise）                     │
│  ├─ 按类型反思（Type-wise）                      │
│  └─ 全局反思（Global）                           │
├─────────────────────────────────────────────────┤
│  执行策略（Strategy）                            │
│  ├─ 规则驱动（Rule-based）                       │
│  ├─ LLM驱动（LLM-driven）                        │
│  └─ 混合驱动（Hybrid）                           │
├─────────────────────────────────────────────────┤
│  迭代模式（Iteration）                           │
│  ├─ 单轮反思（One-pass）                         │
│  ├─ 多轮收敛（Multi-round convergence）          │
│  └─ 双阶段（Two-phase: always → context）        │
└─────────────────────────────────────────────────┘
```

### 1.1 审查粒度

| 粒度 | 适用场景 | 优势 | 局限 |
|------|----------|------|------|
| **按篇反思** | 单篇语境误判、文章特有错误 | 能捕捉局部语境依赖 | 无法发现跨篇系统性错误 |
| **按类型反思** | 同一类错误在全书重复 | 一次修正全部，效率高 | 需要先发现系统性模式 |
| **全局反思** | 跨章交叉验证、一致性审查 | 发现跨章矛盾 | 计算成本高 |

**选择原则**:
- 第一轮用按篇打底，积累错误模式
- 发现系统性模式后用按类型批量修正
- 最后用全局收尾，交叉验证一致性

### 1.2 执行策略

| 策略 | 适用场景 | 优势 | 局限 |
|------|----------|------|------|
| **规则驱动** | 明确规则可枚举、无歧义 | 精确、快速、可解释 | 无法处理语义判断 |
| **LLM驱动** | 需要语义理解和上下文判断 | 灵活、泛化能力强 | 成本高、可能引入新错误 |
| **混合驱动** | 规则预处理 + LLM精修 | 兼具效率与准确性 | 需要精心设计分界线 |

**选择原则**:
- 能用规则就用规则（cheaper & deterministic）
- 规则无法覆盖时才用LLM
- 大规模修正用混合策略（规则先过滤80%，LLM处理剩余20%）

### 1.3 迭代模式

| 模式 | 适用场景 | 修正量趋势 |
|------|----------|------------|
| **单轮反思** | 数据质量已较高，仅需一次复查 | 一次修正即达标 |
| **多轮收敛** | 初始质量较差，需迭代提升 | 指数衰减 |
| **双阶段** | 先批量修正明确错误，再人工审查歧义项 | 阶段1批量，阶段2精细 |

**收敛判断标准**:
- 修正量降至质量基线（如<5%）
- 连续两轮修正量变化<10%
- 新错误模式数=0

## 二、反思工作流的四个阶段

所有反思任务都遵循以下四阶段流程:

```
┌─────────────────────────────────────────────────┐
│  Phase 0: 预备（Preparation）                    │
│  ├─ 定义质量标准与错误模式                       │
│  ├─ 编写SKILL文档（方法论+已知模式）             │
│  └─ 准备检测工具（lint/grep/统计脚本）           │
├─────────────────────────────────────────────────┤
│  Phase 1: 侦察（Reconnaissance）                 │
│  ├─ 抽样审查，发现主要错误类型                   │
│  ├─ 统计错误分布（按篇/按类型）                  │
│  └─ 确定审查策略（按篇/按类型/全局）             │
├─────────────────────────────────────────────────┤
│  Phase 2: 执行（Execution）                      │
│  ├─ Agent批量审查（嵌入SKILL文档）               │
│  ├─ 提取修正建议（JSON格式）                     │
│  └─ 应用修正（脚本自动化）                       │
├─────────────────────────────────────────────────┤
│  Phase 3: 验证（Validation）                     │
│  ├─ 运行lint检查（格式验证）                     │
│  ├─ 对比修正前后统计（修正量/类型分布）          │
│  ├─ 抽样复查（人工spot-check）                   │
│  └─ 决策：收敛结束 or 进入下一轮                 │
└─────────────────────────────────────────────────┘
```

### Phase 0: 预备

**核心产出**: SKILL文档 + lint工具

**SKILL文档必备内容**:
1. 质量标准:什么是"好"的内容？（可量化指标）
2. 已知错误模式:从之前工作中积累的典型错误（表格形式，错误→正确→说明）
3. 审查清单:Agent应检查哪些维度？（格式/边界/类型/一致性）
4. 修正格式规范:JSON输出的schema定义

**lint工具**:
- 格式检查:标注边界完整性、符号闭合
- 统计脚本:错误类型分布、修正量预估

### Phase 1: 侦察

**目标**: 了解当前数据质量现状，确定审查策略

**侦察步骤**:
1. 抽样审查（10-20篇，覆盖不同体裁）
2. 统计错误分布（按篇统计、按类型统计）
3. 确定策略（若某类错误在>50%文章中重复 → 按类型反思）

### Phase 2: 执行

**核心**: Agent批量审查 + 自动化应用修正

**Agent提示词设计**:

```markdown
你是博客内容质量审查员。对文章NNN执行深度反思。

## 输入
- 已处理文件: path/to/file.md
- SKILL文档: skills/SKILL_XX_反思.md（已知错误模式）
- 质量基线: 预期修正量X-Y处

## 任务
1. 读取SKILL文档，了解所有已知错误模式
2. 逐段审查，检查:
   - 格式问题（lint结果）
   - 已知错误模式（SKILL表格）
   - 新发现的问题
3. 输出JSON:
   {
     "corrections": [
       {
         "id": "内容ID",
         "field": "字段名",
         "old_value": "原值",
         "new_value": "新值",
         "reasoning": "修正理由"
       }
     ],
     "new_patterns": ["新发现的错误模式描述"],
     "stats": {
       "total_checked": 数量,
       "corrections_count": 修正数,
       "confidence": "high/medium/low"
     }
   }

## 约束
- 不修改原文字符，只修改标注/分类
- 每处修正必须附理由
- 低于质量基线时，二次复查
```

### Phase 3: 验证

**验证四步**:
1. 格式验证（必须通过）
2. 统计对比（修正前后质量指标提升）
3. 抽样复查（人工spot-check 5-10%）
4. 收敛判断（修正量<5%、新模式=0、lint 100%通过）

## 三、反思的技术架构

### 3.1 数据流

```
原始数据（AI生成）
    │
    ├─ 第一轮反思 ──────────┐
    │   ├─ 修正数据          │
    │   └─ 新错误模式        │
    │                       ▼
    ├─ 更新SKILL文档（模式积累）
    │                       │
    ├─ 第二轮反思 ──────────┘
    │   ├─ 修正数据
    │   └─ 新错误模式（更少）
    │
    ▼（多轮迭代）
高质量数据（发布）
```

### 3.2 关键脚本约定

| 脚本类型 | 命名规范 | 功能 |
|----------|----------|------|
| 侦察脚本 | lint_*.py | 格式检查、错误统计 |
| 提示词生成 | generate_*_prompts.py | 为每篇生成审查提示词 |
| 修正应用 | apply_*_fixes.py | 将JSON修正写入文件 |
| 迁移脚本 | migrate_*.py | 批量类型转换 |
| 验证脚本 | validate_*.py | 修正后质量验证 |

### 3.3 JSON修正格式

```json
{
  "article": "NNN",
  "round": 1,
  "corrections": [
    {
      "id": "内容/实体的唯一ID",
      "type": "错误类型",
      "field": "字段名",
      "old_value": "修正前的值",
      "new_value": "修正后的值",
      "reasoning": "修正理由（必填）",
      "confidence": "high/medium/low",
      "evidence": "支持证据"
    }
  ],
  "new_patterns": [
    {
      "pattern": "错误模式描述",
      "frequency": "估计出现频率",
      "solution": "修正方法"
    }
  ],
  "stats": {
    "total_items": 123,
    "corrections_count": 45,
    "correction_rate": 0.366,
    "quality_score": 0.95
  }
}
```

## 四、反思设计清单

当你为新工序设计反思流程时，按以下清单逐项确认:

### 前置问题
- [ ] 质量标准可量化吗？如何定义"好"的输出？
- [ ] 错误模式可枚举吗？能否列出前10种高频错误？
- [ ] 规则能覆盖多少？有多少错误可用正则/规则检测？
- [ ] LLM必要吗？剩余错误是否必须依赖语义理解？

### 粒度选择
- [ ] 按篇审查是否必要？是否存在文章特有语境依赖？
- [ ] 按类型审查是否可行？是否存在跨篇重复的系统性错误？
- [ ] 全局审查是否需要？是否需要跨篇交叉验证一致性？

### SKILL文档设计
- [ ] 已知错误模式表（格式：错误→正确→说明）
- [ ] 审查清单（Agent应检查哪些维度？）
- [ ] 修正JSON schema（字段定义、必填项、值域约束）
- [ ] 质量基线（每篇预期修正量范围）
- [ ] 收敛标准（何时停止迭代？）

### 工具准备
- [ ] lint脚本（格式检查，返回错误位置）
- [ ] 统计脚本（错误分布、修正量预估）
- [ ] 应用脚本（将JSON修正写入文件）
- [ ] 验证脚本（修正后质量验证）

### 执行计划
- [ ] 侦察阶段:抽样10-20篇，统计错误分布
- [ ] 第一轮:全覆盖审查，积累错误模式
- [ ] 第二轮:基于积累模式，修正量预期减半
- [ ] 第N轮:修正量<5%或新模式=0时停止

## 五、反思的最佳实践

### 5.1 提示词设计原则

1. **嵌入完整SKILL文档** — 不要摘要，全文嵌入
2. **明确质量基线** — 告诉Agent"正常文章预期X-Y处修正"
3. **要求附理由** — 每处修正必须附reasoning字段
4. **要求提取新模式** — 除了修正，还要求Agent输出new_patterns
5. **限制修改范围** — 明确"只改标注/分类，不改原文字符"

### 5.2 迭代控制原则

| 轮次 | 策略 | 质量目标 |
|------|------|----------|
| 第一轮 | 广度优先 | 全覆盖审查，宁可多修也不漏修 |
| 第二轮 | 精度提升 | 修正量预期减半 |
| 第三轮 | 收敛验证 | 修正量应<20%（相对第二轮） |
| 第N轮 | 终止条件 | 修正量<5%或新模式数=0时停止 |

### 5.3 错误模式积累原则

- **表格化沉淀** — 用Markdown表格而非自然语言
- **三列结构** — 错误模式 | 正确形式 | 说明/理由
- **实例优先** — 每种模式附至少1个实际案例
- **分类组织** — 按错误类型分组
- **版本控制** — 每轮反思后更新SKILL文档

### 5.4 质量保障原则

- **lint先行** — 修正前先跑lint，格式错误优先修复
- **抽样复查** — 每轮修正后人工抽查5-10%
- **对比验证** — 修正前后运行统计脚本，质量指标应单调提升
- **回归测试** — 新一轮修正不应破坏旧一轮的成果
- **人机协作** — Agent修正可信度低的（confidence=low），人工二次审查

## 六、总结

反思是将AI生成内容从"能用"提升到"精确"的关键环节。核心洞察:

1. **系统性错误可量化** — 错误模式可被识别和批量修正
2. **Agent可自动化** — 通过SKILL文档嵌入领域知识，Agent可自主完成80-90%的审查修正
3. **知识可沉淀** — 错误模式表、lint工具可在后续文章/工序中复用
4. **质量可收敛** — 多轮迭代的修正量呈指数衰减，3-5轮可达到发布质量

**反思不是简单的"检查-修正"，而是方法论驱动的知识积累过程。** 每一轮反思都在为下一轮、下一工序、下一项目积累可复用的资产。
