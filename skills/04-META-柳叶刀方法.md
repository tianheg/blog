# 元技能04: 柳叶刀方法 — 精细分解与混合解决方案

## Agent 执行摘要

**一句话**：将复杂任务精准分解为独立模块，针对每个子问题选择最优方案（规则/代码/LLM），避免一刀切。
**何时使用**：任务复杂、单一方法（纯 LLM 或纯规则）效果不佳或成本过高时。
**核心步骤**：
1. 按决策树判断每个子任务的性质
2. 选择对应方案：规则查表 → 代码约束 → LLM 语义 → 人工兜底
3. 确保子任务之间 MECE（互斥且完备）
4. 每层保存中间输出，便于调试

## 检查清单

- [ ] 每个子任务有明确输入/输出/验证标准
- [ ] 能用规则就不用 LLM，能用代码就不用规则
- [ ] 子任务之间无重叠、无遗漏（MECE）
- [ ] 每层有中间输出，支持追溯调试
- [ ] 成本和精度达到平衡

## 详细参考

### 任务分工决策树

```
问题分析
    ├─ 是否有明确规则？
    │   ├─ 是 → 【规则+代码】（格式校验、类型过滤）
    │   └─ 否 ↓
    ├─ 是否结构化匹配？
    │   ├─ 是 → 【代码算法】（实体共现、时间对齐）
    │   └─ 否 ↓
    ├─ 是否需要语义理解？
    │   ├─ 是 → 【LLM】（因果推理、情感分析）
    │   └─ 否 ↓
    └─ 是否可安全自动化？
        ├─ 是 → 【规则+LLM 联合】（规则筛选 + LLM 确认）
        └─ 否 → 【人工审查】（保守策略）
```

### 混合解决方案矩阵

| 任务特征 | 推荐方案 | 示例 |
|----------|----------|------|
| 有标准答案的查表任务 | 代码 | 日期查询、常量映射 |
| 可枚举规则的模式匹配 | 规则+Regex | 格式提取、关键词匹配 |
| 需要上下文推理 | LLM | 语义推断、风格判断 |
| 结构化图计算 | 代码 | 关联分析、网络分析 |
| 多源数据一致性检查 | 规则+代码 | 数据验证、约束检查 |
| 歧义消解（有先验） | 规则→LLM→人工 | 命名实体消歧 |
| 语义分类（无标准） | LLM | 内容分类、情感分析 |
| 批量重复任务 | 自动化优先 | 批量处理流水线 |
| 高风险决策 | 人工确认 | 关键业务决策 |

### LLM 与规则的协同模式

**模式 1: 规则前置筛选 + LLM 精炼**
```
全部数据（100万条）
  ↓ [规则筛选]: 共享特征/时间接近
候选数据（5000条） ← 减少 99.5%
  ↓ [LLM 推理]: 语义判断
最终结果（500条）
```

**模式 2: LLM 生成候选 + 规则验证**
```
LLM 生成候选
  ↓ [启发式规则验证]
    - 格式检查
    - 值域约束
安全结果（80%）→ 自动确认
存疑结果（20%）→ 人工审查
```

**模式 3: 规则兜底 + LLM 补全**
```python
def process_data(data):
    # 第一层: 规则查表（40% 覆盖）
    if data in RULE_TABLE:
        return RULE_TABLE[data]
    # 第二层: 代码约束推理（+30% 覆盖）
    if can_infer_by_constraint(data):
        return infer(data)
    # 第三层: LLM 语义推理（+25% 覆盖）
    return llm_infer(data)
    # 人工: 5%（极端歧义）
```

### MECE 分解原则

**Mutually Exclusive, Collectively Exhaustive**（互斥且完备）

检查清单：
- [ ] 子任务之间无重叠（避免重复处理）
- [ ] 子任务覆盖全部场景（无遗漏）
- [ ] 每个模块有明确输入/输出/验证标准

### 反模式（避免）

| 反模式 | 表现 | 后果 | 解决 |
|--------|------|------|------|
| 过度依赖 LLM | 所有任务都用 LLM | 成本爆炸、速度慢、不稳定 | 能用规则就用规则 |
| 规则过度工程 | 为每个边界写规则 | 维护困难、泛化能力差 | 分层策略 |
| 黑箱流水线 | 各模块独立，无中间输出 | 调试困难、误差传播 | 每阶段保存中间文件 |
| 盲目端到端 | 单一大模型解决所有问题 | 可解释性差 | 模块化流水线 |
