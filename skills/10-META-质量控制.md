# 元技能10: 质量控制 — 自动化质检体系与工具链设计

## Agent 执行摘要

**一句话**：用规则脚本自动拦截低级错误（格式、语法、数据完整性），在 Agent 驱动的反思之前建立第一道防线。
**何时使用**：每次数据变更后、Agent 审查前，需要快速、低成本验证时。
**核心步骤**：
1. 按 L1→L5 逐层执行检查，低层级未通过时不执行高层级
2. 输出标准化错误报告（文件、位置、错误类型、建议）
3. 修正后重新运行，直到 100% 通过

**QC vs 反思**：
- QC：规则驱动、秒级、无 LLM 成本、输出 PASS/FAIL
- 反思：语义驱动、分钟级、有 LLM 成本、输出 JSON 修正建议

## 检查清单

- [ ] L1 文本完整性通过（编码正确、文件存在）
- [ ] L2 语法正确性通过（标注边界闭合、格式正确）
- [ ] L3 数据完整性通过（必填字段、值域约束）
- [ ] L4 业务逻辑通过（时序合理、约束一致）
- [ ] L5 跨工序一致性通过（索引与文本一致）
- [ ] 输出报告包含文件路径、行号、错误类型、修复建议

## 详细参考

### 五层质检体系

```
L5: 跨工序一致性（实体索引 ↔ 标注文本一致性）
L4: 业务逻辑验证（时序合理性、约束一致性）
L3: 数据完整性（必填字段、值域约束、枚举值）
L2: 语法正确性（标注边界、JSON 格式、正则匹配）
L1: 文本完整性（编码正确、原文不变、文件存在）
```

### 脚本命名规范

| 类型 | 命名 | 功能 |
|------|------|------|
| 格式/语法检查 | `lint_*.py` | 标注边界、语法 |
| 数据完整性 | `validate_*.py` | 必填字段、值域 |
| 跨工序一致性 | `cross_validate_*.py` | 多源对齐 |

### 输出格式

```
文件: content/til/software/git-rebase.org
❌ [L2-边界完整性] 第42行
   未闭合的标注: #+BEGIN_SRC 无对应 #+END_SRC
   建议: 补全闭合标记
⚠️  [L3-数据完整性] 第67行
   空链接: [[]]
   建议: 删除空链接或补全目标
```

### 执行顺序

```bash
python lint_text_integrity.py      # L1
&& python lint_markdown.py          # L2
&& python validate_data_schema.py   # L3
&& python validate_logic.py         # L4
&& python cross_validate.py         # L5
```

### 质量指标

| 指标 | 定义 | 目标 |
|------|------|------|
| Lint 通过率 | 通过 lint 的文件数/总文件数 | 100% |
| 错误密度 | 错误数/总字符数 | <0.1% |
| 跨工序一致性 | 一致性检查通过项/总检查项 | >95% |

### 本项目的 QC 工具

```bash
# TIL 质量检查（临时文件、空文件、Org 标题、统计分布）
./scripts/til-lint.sh

# Hugo 构建检查
npm run build

# 链接检查（可选）
# 需安装 htmltest 或类似工具
```

---

**核心洞察**：质量控制 ≠ 反思。QC 是规则驱动的快速检查，反思是语义驱动的智能审查。两者互补，不可互相替代。
